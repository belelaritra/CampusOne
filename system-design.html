1. High-Level Architecture

CampusOne follows a classic three-tier architecture — a stateless React SPA, a Django REST API, and persistent storage — extended with a Keycloak identity layer and a Telegram bot sidecar.

Auth Flows
Browser SSO: React → Keycloak (OIDC login) → RS256 JWT → Django KeycloakAuthentication → User provisioned/synced
Bot: Telegram Bot → Django X-Bot-Secret + X-Telegram-Chat-ID → BotAuthentication → User looked up by chat_id

Keycloak as the identity source — Rather than rolling a custom login/register/password-reset system, Keycloak handles all of that out-of-the-box (OIDC, brute-force protection, password policies, token refresh). Django receives only a signed JWT, validates it against the JWKS endpoint (keys cached 5 min, rotated automatically), and provisions a local User row. This keeps Django stateless and lets Keycloak handle token lifecycle.

Two-layer role model — Coarse roles (campus-staff, campus-security) come from Keycloak realm roles and are synced on every request. Fine-grained roles (OutletAdmin, MessAdminProfile) live in the Django DB and are checked per-request with tiny DB hits. This avoids encoding business logic in Keycloak.

Separate Telegram auth backend — The bot never gets a Keycloak token; it uses a shared secret + chat_id. This is safe because the bot is a trusted internal service, not a public client.

Denormalized snapshots on FoodOrder (user_full_name, user_phone_number, user_email) and FoodOrderItem (price) — order records must be immutable after creation even if the user edits their profile or the outlet changes the price. This is standard e-commerce practice.

DB-level UniqueConstraint on LFClaim (status=PENDING) — rather than trusting application-level checks under concurrent traffic, a partial unique index at the DB layer guarantees only one pending interaction per LFItem ever exists. Race conditions handled by catching IntegrityError in the view.

JSONField for tags — tags are a flat list of strings with no cross-item relational queries needed. A JSONField avoids a separate Tag table and join while Django's __icontains still works for search. If tag analytics became more complex, a proper M2M table would be warranted.

5. Design Choices & Trade-offs

SSO via Keycloak vs. Django-native Auth

Identity layer decision

Chosen: Keycloak
✓ Keycloak (chosen)
  • Out-of-the-box OIDC, password policy, brute-force protection
  • Django stays stateless — only validates JWT on each request
  • Key rotation handled automatically via JWKS + PyJWKClient cache (5 min)
  • Roles synced from Keycloak realm to Django on every request
  • Custom login theme possible (campusone Keycloak theme folder exists)
vs
Django Session / SimpleJWT
  • Simpler to set up initially
  • No external dependency
  • But: must build password reset, email verify, brute-force yourself
  • No centralized SSO for future services (bot, mobile app)
  • Harder to add LDAP / institutional SSO later

Two-Layer Role Model vs. Full Keycloak RBAC

Authorization granularity

Chosen: Hybrid
✓ Coarse in Keycloak + Fine in Django DB
  • Keycloak realm roles: campus-staff, campus-security (is_staff, is_security)
  • Fine: OutletAdmin (one-to-one user→outlet), MessAdminProfile (user→hostel) live in Django DB
  • Admin console can change outlet/hostel assignments without touching Keycloak
  • Per-request DB check is fast (indexed FK lookup)
vs
Full Keycloak RBAC (client roles per outlet)
  • Would need a Keycloak client role per outlet (e.g. outlet_admin_aromas)
  • Adding a new outlet requires Keycloak admin action, not just a DB row
  • Harder to manage programmatically from Django admin console
  • More coupling to Keycloak internal concepts

DB-Level Partial Unique Constraint for LF Claims vs. Application Lock

Concurrency in Lost & Found

Chosen: DB Constraint
✓ UniqueConstraint(item, condition=status='PENDING') + select_for_update
  • Only ONE pending claim per item enforced at DB level — can't be bypassed by app bugs or concurrent workers
  • IntegrityError caught in view → clean 409 response
  • select_for_update row-locks the LFItem to prevent double-status updates
  • Correct under multiple Gunicorn workers or async requests
vs
Application-level check only
  • Simple: check if PENDING claim exists before creating
  • Race condition: two requests can both pass the check and both create
  • Would require Redis distributed lock or advisory lock to fix safely
  • More moving parts

Price + User Info Snapshots on Order vs. FK References

Order immutability

Chosen: Snapshot
✓ Snapshot fields on FoodOrder + FoodOrderItem
  • FoodOrderItem.price = snapshot of MenuItem.price at order time
  • FoodOrder.user_full_name/phone/email = snapshot at order time
  • Historical records stay accurate even if user changes profile or outlet changes price
  • Standard e-commerce pattern (Stripe, Shopify all do this)
vs
Always join to live User / MenuItem
  • Simpler schema — no duplicate data
  • But: if outlet admin changes price, all past orders reflect new price
  • If user deletes account, order history loses display data
  • Not suitable for billing/audit trail

GPS Auto-Resolve vs. Free-text Location in Lost & Found

Location data quality

Chosen: Predefined keys + GPS fallback
✓ Predefined campus location keys + GPS haversine nearest-location
  • location_name must be empty OR one of 30+ campus keys — enforced at serializer
  • If user submits GPS lat/lng instead of name → haversine to find nearest campus landmark
  • Consistent location strings → reliable grouping in analytics, suggestions
  • No misspellings ("libary" vs "library") corrupting analytics
vs
Free-text location input
  • User types anything → easier UX
  • But: analytics by location becomes unreliable (infinite variations)
  • Suggestions by location would break
  • Can't do distance calculations without structured coordinates

SQLite (dev) vs. PostgreSQL vs. Redis for Caching

Storage choices

Trade-off noted
Current design and its implications
  • No Redis — JWKS keys are cached in-process (PyJWKClient, 5 min lifespan). Works fine for a single server. Under multi-worker/multi-server setup, each worker has its own JWKS cache — this is fine since JWKS is read-only. But per-user session state (if ever needed) would require Redis.
  • No Celery — Notification dispatch (LFNotification bulk_create, Telegram) is synchronous in the request-response cycle. For a small campus load this is fine. For scale, these should be moved to async tasks.
  • Media files on local FS — Profile photos, menu item images, LF item images are saved to /media/. Fine for a single server. For multi-server or cloud deploy, would need S3/object storage.
  • No pagination on some endpoints (LF notifications capped at 60, suggestions at 10, top_tags at 20) — explicit limits used instead of cursor pagination. Acceptable for a campus-scale system with bounded data.
